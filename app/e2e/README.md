# E2E-тесты воронки (Playwright)

Сквозные проверки пути «заявка → панель». Дополняют unit-тесты (vitest):
драйвят реальный браузер и настоящий раздел «Заявки», ловя то, что unit не видит —
битую вёрстку, неработающий вход, заявку, не доехавшую до панели.

## Что покрыто

- **`funnel.spec.ts` → «Воронка: заявка → панель»** — работает сейчас.
  Заявка с источником «Сайт» доходит до `/panel/leads`, отображается с верным
  статусом/полнотой и не протекает в чужие вкладки-фильтры.
- **`funnel.spec.ts` → «... → форма → панель (WEB2)»** — каркас под будущую
  публичную лид-форму, помечен `test.fixme`. Снять `.fixme`, когда появятся
  страница направления (WEB1) и форма (WEB2) с `data-testid` из спеки.

## Предпосылки

1. Установить раннер и браузер:
   ```sh
   npm install
   npx playwright install chromium   # в этом окружении браузер уже стоит — шаг можно пропустить
   ```
2. Поднять приложение с живой БД (см. корневой README):
   ```sh
   docker compose up
   ```
   Приложение — на `http://localhost:3001`.
3. Задать доступ администратора для входа в панель:
   ```sh
   export E2E_EMAIL="admin@example.com"
   export E2E_PASSWORD="<пароль администратора>"
   # опционально, если сервер на другом адресе:
   export E2E_BASE_URL="http://localhost:3001"
   ```

## Запуск

```sh
npm run e2e         # прогнать все спеки
npm run e2e:ui      # интерактивный режим
npm run e2e:report  # HTML-отчёт последнего прогона
```

## Заметки

- Хелпер `seedSiteLead` пишет заявку прямо в ту же БД (как это сделает форма
  WEB2) и удаляет её после теста — прогоны не копят мусор.
- Селектор карточки — `data-testid="lead-card"` (в `LeadCard.tsx`), поэтому
  проверки не ломаются при изменении вёрстки.
- Как основа синтетического мониторинга: тот же `funnel.spec.ts` можно гонять
  по расписанию против прода (`E2E_BASE_URL=https://smm.oresh.in`) — но тогда
  сидовую заявку создавать не через прямой доступ к БД, а через саму форму.
